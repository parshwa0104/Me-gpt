# You-GPT: A Journey from Overfitting to a Personalized AI

This tutorial documents the steps, problems, and theoretical machine learning concepts applied while building "You-GPT"—a custom GPT model trained on personal WhatsApp chats to mimic the user's conversational style.

---

## 1. The Inference Problem: How to Talk to the Brain?
**The Problem:** We had a `train.py` script that successfully trained the model and saved the weights (`you_gpt_weights.pth`), but no way to actually interact with it as a chatbot.
**The Concept:** Neural network inference. Training teaches the network how to predict the next word, but "Inference" is the act of providing a prompt, predicting a token, appending it to the prompt, and predicting the next one in a continuous loop.
**The Solution:** We created `chat.py`. 
- It loads the architecture from `gpt.py` and the saved weights.
- It takes user input, tokenizes it using `tiktoken`.
- It feeds the tensor into `model.generate()`.
- It takes the newly generated tokens and decodes them back into readable English text.

---

## 2. The Memorization Problem: Overfitting on Small Data
**The Problem:** The model's validation loss started stalling while the training loss plummeted. The model was "overfitting" because it only had a few megabytes of text to learn from. 
**The Concept:** Model Capacity vs. Dataset Size. A model with millions of parameters acts like a massive sponge. If you give it a tiny puddle of data (like a 3MB text file), it absorbs it instantly—meaning it memorizes the exact sentences rather than learning the general rules of grammar and style.
**The Solution:** 
1. **More Data:** We modified `rawfileprocessing.py` to iterate through an entire folder (`raw_chats`) of WhatsApp `.zip` and `.txt` files. It strips out timestamps and system messages using Regex, and merges them into one massive `clean_chat.txt` dataset.
2. **Regularization:** If adding data isn't enough, we discussed increasing `dropout` (randomly turning off neurons during training to prevent reliance on specific memorized pathways) or increasing `weight_decay` in the AdamW optimizer (penalizing excessively large weights).

---

## 3. The Exploding Parameter Problem: Vocabulary Size
**The Problem:** Even with a small transformer (4 blocks, embedding size 256), the model had a massive 54.63 Million parameters. This was way too high for a ~1MB dataset.
**The Concept:** Embedding Tables. We used OpenAI's `tiktoken` (`cl100k_base`), which has a massive vocabulary of 100,277 unique tokens. 
To translate 100k tokens into a 256-dimensional vector space, the input `token_embedding_table` requires (100k * 256) = ~25.6 Million parameters. 
At the very end of the network, the `lm_head` has to translate the 256-dimensional vector back into 100k probabilities, requiring *another* ~25.6 Million parameters. 
The actual "thinking brain" (the transformer blocks) was only ~3 million parameters. The dictionaries were taking up 95% of the model!

**The Solution: "Weight Tying"**
We added a single line of code to `gpt.py`: 
`self.token_embedding_table.weight = self.lm_head.weight`

**The Theory:** 
If the model learns a 256-dimensional mathematical representation of the word "hello" when reading it (Input), it logically makes sense that it should output that *exact same vector* when it wants to speak the word "hello" (Output). 
By forcing the input embedding table and the output projection layer to share the exact same weights:
1. We instantly deleted ~25.6 Million redundant parameters.
2. The shared weights get updated twice as often during training (once on the way in, once on the way out), resulting in much stronger, more stable word representations. 
43. It acts as a massive regularization technique, heavily preventing overfitting without making the chatbot output gibberish. (This exact technique was used by OpenAI in the original GPT-1 and GPT-2 models).

---

## 4. Pre-Training vs Fine-Tuning
**The Problem:** The model was learning slowly, and because it was initialized with random weights, it had to learn English grammar and the user's specific slang simultaneously.
**The Concept:** Transfer Learning. 
**The Solution:** We imported `GPT2LMHeadModel` from HuggingFace. Instead of creating a random 29M parameter model, we downloaded OpenAI's pretrained 124M parameter model. Because it already read the internet and knew perfect English grammar, running `train.py` for 1500 steps exclusively taught it the user's unique formatting and vocabulary without it having to relearn how to form sentences.

---

## 5. Catastrophic Forgetting (Learning Rate)
**The Problem:** Using a high learning rate (`3e-4`) on a pretrained model can violently overwrite the pretrained weights, destroying the model's underlying knowledge of language.
**The Concept:** "Catastrophic Forgetting".
**The Solution:** We lowered the learning rate to `3e-5` for fine-tuning. This "slow and steady" approach preserved the model's English foundation while smoothly adapting to the WhatsApp dataset, hitting a minimum Validation Loss of 4.26 without destroying the pretrained brain.

---

## 6. The "Off-By-One" HuggingFace Shift Bug
**The Problem:** During generation, the model was outputting bizarre fragments of names and seeming to completely skip words.
**The Concept:** Causal Language Modeling labels.
**The Solution:** In custom PyTorch code, you manually shift the labels (predicting `yb` from `xb`). However, HuggingFace's `GPT2LMHeadModel` *automatically shifts labels internally*. By passing `labels=yb` (which was already shifted), we accidentally shifted it twice! The model mathematically learned to predict token `n+2` instead of token `n+1`. Changing it to `labels=xb` fixed the causal mapping instantly.

---

## 7. Out-Of-Distribution Tokenization Bugs
**The Problem:** The model hallucinated gibberish like `arsh Nah:` when responding.
**The Concepts:** 
1. **The Trailing Space BPE Bug:** The prompt ended in `Parshwa Nahar: ` (with a trailing space). Because BPE merges spaces into the subsequent word, a standalone space token `[220]` never appeared after a colon in the training data. The model panicked at the unfamiliar token sequence and hallucinated. Removing the trailing space fixed it.
2. **The Windows Line-Ending Bug:** The model was trained on a Windows text file formatted with CRLF (`\r\n`), but the python script prompted it with `\n`. The token embeddings were entirely different (`[201, 198]` vs `[198]`), causing out-of-distribution hallucinations. Switching the prompt to `\r\n` fixed it.
3. **The Unseen Name Bug:** Prompting the model with `Friend: ` when no one named "Friend" existed in the training data broke the statistical mapping. Pretending to be `Darshan: ` solved it.

---

## 8. Generation Math: Repetition Penalty and Semantic Loops
**The Problem:** The model generated raw characters (like `sehiatauaknaanaaaa`) and later fell into a depression loop (`I am not sure if I am okay_._._.`).
**The Concept:** `repetition_penalty` and `temperature`. 
**The Solution:** 
When we added a `repetition_penalty` of 1.2, it mathematically banned the AI from using common English tokens and even banned it from hitting the Enter key (`\n`), forcing it into character-level gibberish. Removing the penalty allowed it to type properly. 
When it later generated "I am not okay" repeatedly, it was trapped in a "Semantic Loop" because the context window was suddenly flooded with the word "okay", skewing the probabilities. Tweaking the temperature up to `0.7` or adding a tiny `repetition_penalty` (like `1.05`) acts as a soft guardrail against infinite loops while preserving grammatical flow.
